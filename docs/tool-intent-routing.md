# 工具意图识别机制

这份文档只讲一个问题：意图识别阶段，系统是怎么判断一句话应该走普通聊天、普通工具，还是命中 `continue_task`。

它不是完整任务系统设计文档，只解释“给 intent 模型看什么上下文、为什么这么拼、最后怎么命中工具”。

## 1. 总体分流

当前工具意图识别的几类主要分流是：

- 普通聊天、解释、建议、延伸问答：`continue_chat`
- 混乱、断裂、需要先澄清的输入：`continue_chat`
- 明确取数：例如 `ask_weather`
- 明确要在本机执行并产出文件 / 网页 / 文档：`complex_task`
- 没有专门工具，但可以尝试代为执行的泛化现实任务，例如通过 Home Assistant 操作家里设备：`complex_task`
- 明确是在继续刚才那条任务链，不管它还在执行还是已经做完：`continue_task`
- 明确是在追当前任务状态：`query_task_progress`

这里还有一个保守规则：

- 如果输入像 ASR 纠错残片，或者语义不完整、任务对象不清楚、参数不够稳定，宁可先走 `continue_chat` 让主回复模型请用户澄清，也不要误命中执行类工具。
- 如果当前没有专门工具，但用户是在明确要求“替他做一件事”，并且系统可能借助长期记忆、联网服务或外部后台去尝试完成，也可以优先命中 `complex_task`，而不是直接回答“我做不到”。

这里最容易错的是 `continue_task`，因为它不是简单的“再调用一次工具”，而是在判断：

- 用户现在说的这句话，到底是不是在沿着以前那条任务链继续改

## 2. 为什么 `continue_task` 不再要求 plugin_name

`continue_task` 当前只要求模型填写两个字段：

- `task_id`
- `request`

不再要求模型填写 `plugin_name`。

原因很直接：

1. 一旦模型已经选中了某个 `task_id`，后端就能根据这个 task 自己查到对应 plugin。
2. 让模型同时填写 `plugin_name + task_id` 是重复负担，也更容易出现“task 选对了，但 plugin 填错了”的问题。
3. 对模型来说，真正重要的是：
   - 应该续哪条任务链
   - 用户这次补充了什么要求

所以现在 `continue_task` 的语义更收敛：

```json
{
  "task_id": "task_xxx",
  "request": "用户这次新的补充要求"
}
```

## 3. 为什么 `continue_task` 不能只看单个 task

`continue_task` 每次续做，都会新建一条新的 task，并用 `parent_task_id` 串起来。

如果只是把最近可继续的 task 一条条平铺给模型，会有两个问题：

1. 同一条任务链可能重复出现多次  
   模型容易命中过期的旧 `task_id`

2. 只有最新摘要，没有连续语义  
   用户如果按“最开始那件事”来提，模型容易缺少原始语义锚点

所以当前做法不是“按 task 行给候选”，而是“按任务链给摘要”。

## 4. continue_task 的任务链摘要长什么样

当前每条候选不是变量堆叠，而是自然语言链式摘要。

每条摘要至少包含：

- `latest_task_id`
- 初始任务需求
- 中间轮次对话
- 任务最后回答

也就是说，模型看到的是“这件事一路怎么做过来”的摘要，而不是一堆零散字段。

## 5. 两种典型 case

### Case 1：只有根任务，没有后续续做

用户最开始说：

- “帮我做一个关于天气的小游戏”

任务完成后，候选摘要可能是：

```text
[latest_task_id=task_100]
初始任务需求：帮我做一个关于天气的小游戏
中间轮次对话：无
任务最后回答：已经做出一个可玩的天气小游戏网页。
```

如果用户后面说：

- “那个天气小游戏再加个音效”

模型应该命中：

- `continue_task`
- `task_id=task_100`
- `request=再加个音效`

### Case 2：已经有多轮续做

根任务：

- “帮我做一个关于天气的小游戏”

后续两轮：

- “加一点动画”
- “再炫酷一点”

这时候候选摘要会折叠成一条：

```text
[latest_task_id=task_130]
初始任务需求：帮我做一个关于天气的小游戏
中间轮次对话：
- 任务执行器：第一版小游戏已经完成。
- 用户追加输入：加一点动画
- 任务执行器：已经补上基础动画。
- 用户追加输入：再炫酷一点
- 任务执行器：当前版本已经加入更强的动画效果和视觉强化。
任务最后回答：当前版本已经加入更强的动画效果和视觉强化。
```

如果用户这次说：

- “刚刚那个天气小游戏再加个音效”

模型应该命中：

- `continue_task`
- `task_id=task_130`

这里命中的不是根任务 `task_100`，而是这条链当前最新的可继续节点 `task_130`。

### Case 3：任务还在执行，用户中途补充要求

根任务：

- “帮我做一个关于天气的小游戏”

当前这条链最新节点还在执行中，摘要可能是：

```text
[latest_task_id=task_180]
初始任务需求：帮我做一个关于天气的小游戏
中间轮次对话：
- 任务执行器：第一版小游戏已经完成。
- 用户追加输入：再加一点音效
- 任务执行器：Claude 正在补音效。
任务最后回答：Claude 正在补音效。
```

如果用户这次说：

- “把刚刚那个小游戏改成下雨天更明显一点”

模型仍然应该命中：

- `continue_task`
- `task_id=task_180`
- `request=把刚刚那个小游戏改成下雨天更明显一点`

但如果用户说的是：

- “刚刚那个任务现在做到哪了”

这就不是接续，而是：

- `query_task_progress`

## 6. 为什么“用户追加输入”直接用 task.input

当前任务链里的每个续做节点，本来就有自己的 `task.input`。

这些 `input` 正是：

- 用户为什么又继续提了一次新要求
- 这条任务链后来具体改过什么

所以把这些 `task.input` 直接组织成：

- `用户追加输入：...`

是合理的，而且比只给模型一个 `recent_followups=...` 的平铺字段更自然。

## 7. 当前的收敛规则

当前这套机制先故意收敛成：

- 每条任务链只展示一个候选摘要
- 候选只对应这条链“当前最新的可继续节点”
- 可继续节点当前包括 `accepted / running / completed`
- `failed / canceled / superseded` 默认不进入 `continue_task` 候选
- `continue_task` 工具输入里只保留 `task_id + request`
- 如果用户输入混乱、不完整、像 ASR 纠错残片，当前优先回到 `continue_chat` 做澄清，而不是硬命中 `continue_task`
- 如果用户只是在追当前进度，而不是补充新要求，优先命中 `query_task_progress`

这样做的目标，是让模型稳定地区分两件事：

- 继续补充同一条任务链：`continue_task`
- 只是问现在做到哪：`query_task_progress`
