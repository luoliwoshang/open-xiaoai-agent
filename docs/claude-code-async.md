# Claude Code 作为异步任务执行器的接入说明

这份文档只讲一件事：当前 Claude Code 是怎么作为异步任务执行器接进系统里的。

它不重复解释整个异步任务总线本身，而是聚焦在 Claude 这一条实现链路：怎么受理、怎么执行、怎么汇报进度、怎么续做之前的任务。

关于 Claude 任务产物怎么接入 task artifact，请看单独文档：

- `docs/claude-code-artifacts.md`

## 设计定位

在当前架构里，Claude Code 不是主流程本身，也不是语音入口，而是后台执行层的一种具体实现。

可以简单理解成：

- 小爱负责和用户沟通
- 主流程负责判断要不要派任务、什么时候做任务结果汇报
- Claude Code 负责在后台真正把复杂任务做掉

这个边界很重要，因为它意味着：

- 将来可以替换成别的执行器
- 主流程不需要为了某一个执行器重写
- Claude 只需要做好“任务执行”和“上下文恢复”

## 1. Claude 接入后的整体结构

当前 Claude 这条链路大致由四部分组成：

1. 复杂任务受理层
2. Claude 执行器
3. Claude 私有状态层
4. 任务续做恢复层

它们分别负责：

- 复杂任务受理层：把用户一句话变成异步任务规格
- Claude 执行器：真正调用 `claude` CLI 干活
- Claude 私有状态层：保存 Claude 会话恢复所需的信息
- 任务续做恢复层：让后续补充要求能在原 Claude 上下文中继续推进

## 2. Claude 是怎么被主流程调起来的

当主流程识别出一句话应该走复杂任务时，不会直接把整件事塞进主对话线程里做完，而是会先受理，再后台执行。

Claude 在这个时刻承担的角色是：

- 接收一份已经确定要异步执行的任务请求
- 立刻返回“任务已受理”的前台回执
- 真正执行动作转入后台

也就是说，对用户来说：

- 前台先听到“收到，我先去处理”
- 后台才开始让 Claude 干活

## 3. Claude 对应的任务受理形态

当前复杂任务受理后，会生成一份异步任务规格。

这份规格至少包含这些核心信息：

- 任务属于复杂任务这一类
- 任务标题来自用户请求
- 原始输入会被完整保留
- 后台真正的运行函数交给 Claude 执行器

所以 Claude 不是直接嵌进主流程里的专用分支，而是挂在统一异步任务接口后面。

这层抽象的好处是：

- 主流程只需要知道“有一个后台执行器要跑”
- 不需要关心 Claude 的命令行细节

## 4. Claude 如何启动一个新任务

当前新任务命令形态是：

```sh
claude --dangerously-skip-permissions --print --output-format stream-json --verbose "<task>"
```

这里传给 Claude 的并不是原始用户句子本身，而是经过一层任务包装后的提示词。

当前包装重点约束了几件事：

- 执行过程中要持续汇报阶段性进度
- 进度汇报要简短
- 不要输出太多影响 TTS 的 Markdown 噪音或特殊符号
- 任务未完成时不要提前说已经完成
- 如果有要交付给系统的文件，必须统一放在任务专属目录 `.open-xiaoai-agent/deliverables/<task_id>/` 下
- 面向用户的进度和最终总结，不要暴露工作目录、路径、manifest、命令行等内部工程细节
- 面向用户的最终总结里，也不要直接说“保存为 xxx.png / xxx.html / xxx.txt”
- 面向用户的措辞尽量按普通人能直接听懂的方式表达，不要过于专业和冗长
- 最终总结要尽量说明：
  - 完成了什么
  - 用户接下来怎么用

这背后的核心目标不是让 Claude “更会写 prompt”，而是让 Claude 的输出更适合：

- 被系统持续跟踪
- 被前端展示
- 被语音系统自然播报

## 5. Claude 的流式输出怎么转成任务进度

Claude 当前通过 `stream-json` 输出执行过程。

执行器会逐行读取这些流式消息，并把它们转换成任务系统可理解的信息。

当前主要处理三类消息：

### 5.1 会话初始化消息

当 Claude 建立会话并返回会话标识时，系统会：

- 保存这次会话的恢复信息
- 记录一条“Claude 会话已建立”的事件

这样后面续做任务时，系统才知道该接回哪一个 Claude 上下文。

### 5.2 中间过程消息

当 Claude 在执行过程中持续输出文本时，系统不会原样整段塞给任务层，而是会做两步处理：

1. 清洗文本
2. 抽出一句尽量适合展示和播报的阶段摘要

然后再把这句摘要回写给通用任务层，作为当前进度。

这样做的好处是：

- 前端看到的是清晰的阶段进展
- 语音结果汇报时可以直接复用
- 执行器自己的原始输出和任务层的用户可读摘要可以分开

### 5.3 最终结果消息

当 Claude 返回最终结果时，系统会：

- 保存最终结果
- 把结果交回通用任务层
- 由通用任务层把任务切到完成状态

如果 Claude 返回错误，或者命令执行失败，则任务会被标记为失败。

## 6. Claude 私有状态为什么单独保存

Claude 这条链路里，有一些状态只对 Claude 自己有意义，比如：

- 当前会话标识
- 原始任务提示
- 工作目录
- 最后一次阶段摘要
- 最近一次完整中间文本
- 最终结果
- 错误信息

这些数据不会直接塞进通用任务层，因为那样会带来两个问题：

- 主任务模型会被某一个执行器的私有字段污染
- 将来接入其他执行器时，通用层会越来越臃肿

所以当前做法是：

- 通用任务层只保留大家都需要的共性信息
- Claude 的恢复和运行细节放在 Claude 私有状态层里

## 7. Claude 如何续做之前的任务

Claude 的续做不是“重新再跑一次类似任务”，而是“在同一个执行上下文里接着做”。

当前续做命令形态是：

```sh
claude --dangerously-skip-permissions --resume "<session_id>" --print --output-format stream-json --verbose "<follow-up request>"
```

这条链路的关键点有八步：

1. 用户提出一个补充要求，比如“刚刚那个网页再炫酷一点”。
2. 主流程识别出这不是全新任务，而是续做之前那个任务。
3. 系统找到原任务对应的执行器归属。
4. 系统为这次续做创建一条新的异步任务记录。
5. 新任务会保留自己的新任务 ID。
6. 新任务同时会挂回原任务，形成父子关系。
7. Claude 执行器从私有状态里取出原会话标识。
8. 续做请求通过 `--resume` 回到原 Claude 会话继续执行。

这个设计的重点是：

- 不覆盖旧任务
- 不丢失原上下文
- 让“第一次做”和“后续追加修改”都可追踪

## 8. 为什么 Claude 适合作为第一种执行器

Claude Code 当前作为第一种异步任务执行器，有几个现实优势：

- 本身就适合处理多步骤、本地落地产出的复杂任务
- 支持流式输出，天然适合阶段性进度回传
- 支持会话恢复，能很好承接“继续做刚刚那个任务”这种需求

这让它非常适合拿来验证这条链路：

- 用户自然说一句话
- 系统受理复杂任务
- 后台持续执行
- 中途能看进度
- 完成后能做结果汇报
- 后面还能继续追着同一个任务改

## 9. 当前 Claude 接入的边界

虽然 Claude 这条链路已经能支撑完整的异步任务闭环，但当前仍然有一些边界：

- Claude 是当前的第一种执行器实现，不代表系统只服务 Claude
- 续做能力依赖 Claude 会话恢复能力
- 输出已经做了适配，但仍然要持续约束它更适合语音播报
- 后续如果接入更多执行器，需要各自实现自己的私有状态和恢复逻辑

## 10. 对后续执行器接入的启发

Claude 这条实现验证了一个重要方向：

- 主流程负责调度
- 任务总线负责承接
- 执行器负责干活
- 私有状态负责恢复

如果以后接入 Codex、OpenClaw、OpenCode，比较理想的方式也应该是沿着这条边界继续扩展，而不是把执行器逻辑重新塞回主流程里。
