# XiaoAiAgent 当前功能总览

这篇文档是给产品梳理、页面设计和示意图绘制用的。

目标只有两个：

- 把当前仓库里已经实现的能力列清楚
- 把“现在能做什么”和“现在还没做什么”分开，避免画图时混淆

## 1. 产品定位

`XiaoAiAgent` 是一个挂在 `open-xiaoai` 设备 / client 后面的独立服务端。

它不是单纯的问答后端，而是一个把“小爱同学”增强成可对话、可派任务、可查进度、可继续追任务、可通过 IM 触达用户的编排服务。

可以把它理解成 5 层能力：

1. 语音入口层
2. 对话与调度层
3. 工具与异步任务层
4. IM 触达层
5. Dashboard 调试层

## 2. 当前已经实现的功能

### 2.1 语音入口与主对话

- 接收 `open-xiaoai` client 通过 WebSocket 转发的最终 ASR 文本
- 在收到 ASR 结果后，可选择中断原始小爱流程，再由当前服务接管回复
- 支持普通对话，不只是一次性问答
- 回复通过设备侧现有播放链路回到音箱 / client
- `reply` 模型支持流式输出
- `intent` 和 `reply` 可以并行 speculative 执行，减少普通聊天路径等待时间

### 2.2 上下文记忆

- 会话上下文会持久化保存
- 当前是滑动窗口策略
- 规则是 `last_active + session.window_seconds`
- 默认窗口为 `300` 秒
- 可以在 Dashboard 调试台里调整会话窗口秒数

上下文里现在会记录：

- 用户输入
- 助手回复
- 在真正播报成功后的任务结果汇报内容

需要特别区分：

- 这一部分是“最近几分钟的会话窗口”
- 不是下面要说的“长期记忆”

### 2.3 长期记忆

- 当前已经支持主流程长期记忆
- 默认 memory key 是 `main-voice`
- reply 会参考长期记忆
- `complex_task` / `continue_task` 也会拿到这份长期记忆
- intent 当前不会读取长期记忆
- 长期记忆不会按每轮对话高频写回，而是在一次 session 自然结束后再做低频整理
- 记忆整理被约束为禁止凭空补全未出现事实；如果新对话与旧记忆冲突，应更新旧记忆
- 默认实现是本地 Markdown 文件，不是向量库
- 记忆目录配置保存在 MySQL `settings.memory_storage_dir`
- 记忆更新日志会写入 MySQL，供 Dashboard 查看 diff

这部分的实现边界和 dashboard 能力，单独见：

- `docs/memory.md`

### 2.4 意图识别与工具路由

- 支持把用户输入路由成普通聊天、工具调用、异步任务、继续任务、查询任务等不同路径
- 工具结果通常不会原样机械播报，而是经过 `reply` 模型整理后再回复
- `continue_task` 当前不是看单条 task，而是看按任务链折叠后的自然语言摘要
- `continue_task` 当前只要求命中 `task_id` 和新的补充要求，插件名由后端按 task 自己解析
- 当前内置工具包括：
  - `ask_weather`
  - `list_tools`
  - `complex_task`
  - `query_task_progress`
  - `cancel_task`
  - `continue_task`

工具意图识别、尤其是 `continue_task` 的链式摘要拼接方式，可参考：

- `docs/tool-intent-routing.md`

### 2.5 异步任务系统

- 支持受理复杂、耗时较长的任务
- 任务状态目前包括：
  - `accepted`
  - `running`
  - `completed`
  - `failed`
  - `canceled`
- 复杂任务不会阻塞当前正常对话
- 用户可以在日常对话中继续正常聊天
- 用户可以查询任务进度
- 用户可以继续追加要求，命中原任务上下文
- 用户可以取消任务

当前异步任务系统的几个关键体验点：

- 任务完成后不会强行立刻打断当前对话
- 系统会把任务完成标记为待汇报
- 当前语音通道空闲时，系统会主动把待汇报结果告诉用户；如果当前正忙，就等安全时机再汇报
- 进度查询会带上任务标题、真实状态、当前摘要，而不只是摘要文本

### 2.6 Claude Code 作为异步执行器

- 当前 `complex_task` / `continue_task` 已接入 Claude Code CLI
- 支持新任务执行
- 支持基于 Claude `session_id` 的继续执行
- Claude 私有状态和主任务表是分开的
- continuation 会创建新的 task 行，并通过 `parent_task_id` 关联，而不是覆盖旧任务

### 2.7 任务产物交付

- 当前任务可以挂接 artifact
- artifact 属于当前 `task_id`
- `completed` 后不再允许继续新增 artifact
- 插件与任务系统之间不直接以本地路径作为边界协议
- 当前任务 artifact 会缓存到本地目录，并把元数据持久化到 MySQL
- 只要 artifact 被登记进任务系统，就会自动创建对应的产物交付记录
- 任务结果汇报前，主流程会先尝试把这些产物发到当前默认 IM 渠道
- 如果当前没有可用渠道，结果汇报会明确说明“当前没有可用的渠道发送产物”
- 如果发送成功，结果汇报会自然提示产物已经发到对应渠道
- Dashboard 调试台支持直接下载某个任务对应的 artifact

### 2.8 Claude Code 产物 manifest 机制

- Claude 任务已经支持通过 manifest 声明交付产物
- 所有可交付文件必须统一落在任务专属目录 `.open-xiaoai-agent/deliverables/<task_id>/`
- manifest 只负责声明：
  - 哪些文件要交付
  - 文件名
  - 类型
  - MIME 信息
- runner 会读取 manifest，再把对应文件导入任务 artifact 系统

这意味着：

- Claude 可以在自己的工作目录里自由产出文件
- 但交给主系统的是规范化 artifact，而不是裸本地路径

### 2.9 IM Gateway：微信一期能力

当前 IM 网关是独立于 OpenClaw 的第一期能力，重点是“出站触达”。

已经实现的能力：

- 微信扫码登录
- 扫码成功后前端确认添加账号
- 如果当前还没有默认投递配置，确认添加后会自动把这个账号和扫码用户写成默认调试渠道
- 多微信账号管理
- 触达对象管理
- 为每个账号选择默认触达对象
- 配置“小爱回复镜像到默认微信目标”
- 默认渠道调试发送文本
- 默认渠道调试发送图片
- 默认渠道调试发送文件
- 调试发送选择图片或文件后，前端会显示已选内容的预览或文件信息

需要特别区分：

- 自动镜像：当前只支持文本
- 手动调试发送：支持文本 / 图片 / 文件

### 2.10 微信适配层媒体能力

微信适配层当前已经支持：

- `SendText`
- `SendImage`
- `SendFile`

其中：

- 图片发送已经按 `openclaw-weixin` 官方实现对齐
- 图片走 `getuploadurl -> CDN 上传 -> sendmessage(image_item)`
- 文件也走上传后再发送消息体的链路

### 2.11 Dashboard 调试控制台

当前前端是 React + Vite Dashboard。

这里的定位要非常明确：

- 它不是面向普通用户的日常操作台
- 它是给开发、联调、验证、排障使用的调试控制台
- 它承担的是“看状态、打调试入口、看日志、查任务、查交付”的职责

已经有 3 个主要页面：

- Dashboard 调试首页
- 长期记忆页
- Settings 调试设置页
- Logs 调试日志页

#### Dashboard 调试首页

- 手动把一段识别文本直接送进主流程做调试
- 展示任务列表
- 以弹窗形式查看当前选中任务详情
- 查看任务事件流
- 查看任务 artifact
- 查看 Claude 相关记录
- 查看最近会话内容
- 观察当前主流程是否忙碌、是否存在真实小爱连接
- 在左下角状态区直接查看 IM 默认渠道是否已配置、是否已启用，以及当前命中的账号 / 目标摘要

#### Settings 调试设置页

当前分成系统设置与触达配置两大域。

系统设置：

- 会话窗口秒数设置
- 长期记忆目录设置

#### 长期记忆页

- 查看 `main-voice` 当前记忆文件
- 手动编辑并保存长期记忆正文
- 查看记忆更新日志
- 用 diff 风格查看某次更新前后的变化
- 区分系统按 session 收尾生成的更新，与手动编辑产生的更新

触达配置：

- 默认触达 / 镜像规则
- 手动调试发送
- 微信账号管理
- 触达对象管理

#### Logs 调试日志页

- 查看后端运行日志
- 支持分页
- 支持切换每页条数
- 第 1 页自动刷新
- 紧凑展示日志级别、时间、source、消息正文和原始日志

### 2.12 后端日志系统

- 后端标准 logger 输出会被持久化到 MySQL
- 日志会保留：
  - 时间
  - 推断级别
  - source 文件与行号
  - 解析后的 message
  - 原始 raw 日志行
- IM 关键链路已补 runtime log，便于排查：
  - 微信登录
  - 调试发送文本 / 图片 / 文件
  - 镜像入队
  - adapter 请求阶段
  - 投递成功 / 失败

### 2.13 运行时设置与重置

- 会话窗口设置保存在 MySQL
- IM 默认触达配置保存在 MySQL
- 长期记忆目录保存在 MySQL
- 服务启动会确保默认设置存在
- 当前还提供运行时重置接口：
  - 清空会话等运行时状态

### 2.14 数据持久化

当前项目使用 MySQL 持久化以下信息：

- 任务
- 任务事件
- 任务 artifact 元数据
- 会话历史
- Claude 私有状态
- 运行时设置
- IM 账号 / 目标 / 事件
- 后端运行日志
- 长期记忆更新日志

本地磁盘缓存当前用于：

- 任务 artifact 文件缓存
- IM 调试发送图片 / 文件缓存
- 长期记忆 Markdown 文件

## 6. 现在明确还没做的事情

这部分很重要，画图时不要画成“已完成”。

- 还没有 IM 入站会话
- 还没有微信群聊路由
- 还没有自动 IM 图片镜像或文件镜像
- 还没有把 Codex / OpenClaw / OpenCode 等更多执行器接进主流程
- 还没有完整的任务产物投递状态追踪
- 还没有父 task / 根 task 的 artifact 聚合视图
- 还没有更完整的语音打断检测
