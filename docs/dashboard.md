# Dashboard 调试定位说明

这份文档只讲 Dashboard 目前对主服务提供了哪些调试入口，以及这些入口和主流程之间的关系。

先明确一个定位：

- Dashboard 不是面向普通用户的日常工作台
- Dashboard 是 XiaoAiAgent 的调试与排障控制台
- 它的职责是帮助开发者或操作者验证主流程、观察运行状态、排查问题、手动触发调试动作

## 1. 手动送入识别文本

Dashboard 首页现在提供了一个“手动送入一条识别文本”的输入区。

它的目标不是替代真实设备上的 ASR，而是提供一个稳定的调试入口，用来验证：

- intent 路由是否符合预期
- 普通 reply / tool / async task 是否按主流程执行
- 服务端主流程在没有真实设备时是否仍然可调试
- 主流程忙闲控制是否生效

## 2. 小爱连接状态

Dashboard 还会展示当前 XiaoAI client 和服务端之间的 WebSocket 连接状态。

当前信息至少包括：

- 是否已经连接
- 当前活跃连接数
- 最近一次连接时间
- 最近一次断开时间
- 最近一次连接来源地址

后端接口：

- `GET /api/xiaoai/status`

同时，这块状态也会进入 `GET /api/state` 的 `xiaoai` 字段，方便前端在统一轮询里直接展示。

## 3. IM 默认渠道状态

Dashboard 左下角的状态区，当前除了小爱连接与主语音通道状态，还会展示 IM 默认渠道状态。

这里主要回答三个调试问题：

- 当前有没有已经选好的默认 IM 渠道
- 这个默认渠道现在是已启用、已配置但未启用，还是已经失效
- 当前命中的账号 / 触达对象大致是谁

当前展示逻辑基于 `GET /api/state` 里的：

- `settings.im_delivery_enabled`
- `settings.im_selected_account_id`
- `settings.im_selected_target_id`
- `im.accounts`
- `im.targets`

也就是说，这里不要求额外的单独接口，前端会直接用统一轮询状态判断：

- `IM 默认渠道已启用`
- `IM 默认渠道已配置`
- `IM 尚未选定默认渠道`
- `IM 默认渠道配置失效`
- `IM 未配置渠道`

这样操作者在首页不进入设置页，也能立刻判断默认 IM 触达链路当前是不是可用。

任务列表中的详情查看，当前也改成了“点击任务后弹窗展开”的方式。
这样调试首页主布局更稳定，任务状态、事件、产物和执行器信息会在独立弹窗里集中查看。

同时，`近期对话` 仍然保留在主界面，作为全局上下文区域单独展示，不和任务弹窗混在一起。

## 4. 长期记忆页

Dashboard 现在还有一个单独的 `长期记忆` 页面。

它的职责不是做通用知识库管理，而是给当前默认文件型记忆实现提供调试入口。

当前可以做的事情：

- 查看 `main-voice` 对应的长期记忆正文
- 手动编辑并保存记忆文件
- 查看记忆更新日志
- 对于 `session_summary` 日志，可以展开看到本次总结所依据的原始对话消息
- 用 diff 风格回看某次变更前后

这个页面直接对应底层记忆文件本身，不会预先塞入默认模板内容。

当前这些更新日志主要来自两类来源：

- 一次主语音 session 自然结束后的系统整理
- dashboard 上的手动保存

这里要注意：

- 这些“文件管理 / 日志查看”能力属于当前 `filememory` 实现者
- 不是抽象记忆接口本身的职责

当前相关接口：

- `POST /api/settings/memory`
- `GET /api/memory/file`
- `POST /api/memory/file`
- `GET /api/memory/logs`

## 5. 后端接口

当前接口为：

- `POST /api/assistant/asr`

请求体：

```json
{
  "text": "帮我继续刚刚那个网页任务"
}
```

成功返回：

```json
{
  "ok": true,
  "text": "帮我继续刚刚那个网页任务"
}
```

## 6. 这个入口到底做了什么

这个入口会把 Dashboard 输入的文本，当成一条“已经识别完成的用户文本”，
送进 assistant 的主流程，但它不会依赖真实 XiaoAI 设备连接。

当前行为是：

1. 使用固定的主语音 `historyKey`：`main-voice`
2. 使用服务端内置的 debug voice channel
3. 按正常主流程继续执行 `intent` / `reply` / tool / async task
4. 把“准备播放”和“播放文本”写入后端日志，而不是发到真实设备

也就是说，它模拟的是：

- 一条稳定可复现的 ASR 输入
- 一条不依赖设备在线的调试播报通道

同时它和真实小爱入口会共享同一段主语音上下文，所以像“继续刚刚那个任务”这类追问，也会落在同一段历史里。

## 7. 什么时候会失败

当前至少有三类明确失败：

1. 文本为空
   - 返回 `400`

2. 当前主语音通道正忙
   - 返回 `409`
   - 表示当前已经有一轮会发声的 assistant 主流程正在执行

当前版本下，这个入口不再因为“没有最近设备通道”而失败。
哪怕小爱当前没有连接，也可以继续做服务端主流程调试。

## 8. 为什么不是直接新建一条设备会话

当前这个调试入口的重点是：

- 验证主流程
- 验证服务端独立调试能力
- 验证忙闲控制

而不是模拟整条设备接入链路，或者偷偷复用当前在线的小爱连接来完成调试播报。

所以它刻意把两件事拆开：

- 上下文统一写进固定的 `main-voice` 会话历史
- 调试播报走独立的 debug voice channel

这样既能共享真实语音对话上下文，又不会把“是否有设备在线”变成 debug 主流程的前置条件。

## 9. 和记忆系统的关系

当前 dashboard 调试入口和长期记忆的关系是：

- debug ASR 仍然写入主语音历史 `main-voice`
- reply 会按 `main-voice` 读取长期记忆
- intent 也会按 `main-voice` 读取长期记忆
- `complex_task` / `continue_task` 也会拿到这份长期记忆

所以从 dashboard 里手动送入一句文本，本质上是在调试：

- 主流程路由
- 主语音历史
- 长期记忆参与 intent / reply / task 的实际效果

## 10. 和单通道语音调度的关系

当前 assistant 仍然遵守“同一时刻只允许一个会发声主流程运行”的规则。

因此：

- 如果真实设备上的主流程正在播报，Dashboard 送入的文本会被拒绝
- 如果 Dashboard 刚送入了一条文本，新的设备 ASR 也会因为通道忙而被忽略

这个入口是主流程的一部分，不是绕过主流程的旁路。
