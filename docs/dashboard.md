# Dashboard 技术说明

这份文档只讲 Dashboard 目前对主服务提供了哪些调试入口，以及这些入口和主流程之间的关系。

## 1. 手动送入识别文本

Dashboard 首页现在提供了一个“手动送入一条识别文本”的输入区。

它的目标不是替代真实设备上的 ASR，而是提供一个稳定的调试入口，用来验证：

- intent 路由是否符合预期
- 普通 reply / tool / async task 是否按主流程执行
- 最近一次可用语音通道是否还能正常播报
- 主流程忙闲控制是否生效

## 2. 后端接口

当前接口为：

- `POST /api/assistant/asr`

请求体：

```json
{
  "text": "帮我继续刚刚那个网页任务"
}
```

成功返回：

```json
{
  "ok": true,
  "text": "帮我继续刚刚那个网页任务"
}
```

## 3. 这个入口到底做了什么

这个入口不会自己创建新的设备连接，也不会模拟一个全新的 XiaoAI session。

当前行为是：

1. 读取 assistant 最近一次成功进入主流程时保留下来的：
   - `historyKey`
   - `voice channel`
2. 把 Dashboard 输入的文本，当成一条“已经识别完成的用户文本”
3. 直接送入主流程的 `HandleUserText(...)`

也就是说，这个入口复用的是：

- 最近一次还能说话的语音通道
- 最近一次还能写入会话历史的上下文

它不是一个完全独立的新会话。

## 4. 什么时候会失败

当前至少有三类明确失败：

1. 文本为空
   - 返回 `400`

2. 当前主语音通道正忙
   - 返回 `409`
   - 表示当前已经有一轮会发声的 assistant 主流程正在执行

3. 最近还没有可复用的语音通道或上下文
   - 返回 `503`
   - 这通常意味着系统还没有接到过一轮真实可用的语音输入

## 5. 为什么不是直接新建一条设备会话

当前这个调试入口的重点是：

- 验证主流程
- 验证语音通道复用
- 验证忙闲控制

而不是模拟整条设备接入链路。

所以它刻意只复用“最近一次可用的 voice channel + conversation context”，而不是额外伪造一层设备 session。这样边界更清楚，调试结果也更贴近真实主流程。

## 6. 和单通道语音调度的关系

当前 assistant 仍然遵守“同一时刻只允许一个会发声主流程运行”的规则。

因此：

- 如果真实设备上的主流程正在播报，Dashboard 送入的文本会被拒绝
- 如果 Dashboard 刚送入了一条文本，新的设备 ASR 也会因为通道忙而被忽略

这个入口是主流程的一部分，不是绕过主流程的旁路。
