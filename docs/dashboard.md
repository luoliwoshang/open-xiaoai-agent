# Dashboard 技术说明

这份文档只讲 Dashboard 目前对主服务提供了哪些调试入口，以及这些入口和主流程之间的关系。

## 1. 手动送入识别文本

Dashboard 首页现在提供了一个“手动送入一条识别文本”的输入区。

它的目标不是替代真实设备上的 ASR，而是提供一个稳定的调试入口，用来验证：

- intent 路由是否符合预期
- 普通 reply / tool / async task 是否按主流程执行
- 服务端主流程在没有真实设备时是否仍然可调试
- 主流程忙闲控制是否生效

## 2. 小爱连接状态

Dashboard 还会展示当前 XiaoAI client 和服务端之间的 WebSocket 连接状态。

当前信息至少包括：

- 是否已经连接
- 当前活跃连接数
- 最近一次连接时间
- 最近一次断开时间
- 最近一次连接来源地址

后端接口：

- `GET /api/xiaoai/status`

同时，这块状态也会进入 `GET /api/state` 的 `xiaoai` 字段，方便前端在统一轮询里直接展示。

## 3. 后端接口

当前接口为：

- `POST /api/assistant/asr`

请求体：

```json
{
  "text": "帮我继续刚刚那个网页任务"
}
```

成功返回：

```json
{
  "ok": true,
  "text": "帮我继续刚刚那个网页任务"
}
```

## 4. 这个入口到底做了什么

这个入口会把 Dashboard 输入的文本，当成一条“已经识别完成的用户文本”，
送进 assistant 的主流程，但它不会依赖真实 XiaoAI 设备连接。

当前行为是：

1. 使用固定的主语音 `historyKey`：`main-voice`
2. 使用服务端内置的 debug voice channel
3. 按正常主流程继续执行 `intent` / `reply` / tool / async task
4. 把“准备播放”和“播放文本”写入后端日志，而不是发到真实设备

也就是说，它模拟的是：

- 一条稳定可复现的 ASR 输入
- 一条不依赖设备在线的调试播报通道

同时它和真实小爱入口会共享同一段主语音上下文，所以像“继续刚刚那个任务”这类追问，也会落在同一段历史里。

## 5. 什么时候会失败

当前至少有三类明确失败：

1. 文本为空
   - 返回 `400`

2. 当前主语音通道正忙
   - 返回 `409`
   - 表示当前已经有一轮会发声的 assistant 主流程正在执行

当前版本下，这个入口不再因为“没有最近设备通道”而失败。
哪怕小爱当前没有连接，也可以继续做服务端主流程调试。

## 6. 为什么不是直接新建一条设备会话

当前这个调试入口的重点是：

- 验证主流程
- 验证服务端独立调试能力
- 验证忙闲控制

而不是模拟整条设备接入链路，或者偷偷复用当前在线的小爱连接来完成调试播报。

所以它刻意把两件事拆开：

- 上下文统一写进固定的 `main-voice` 会话历史
- 调试播报走独立的 debug voice channel

这样既能共享真实语音对话上下文，又不会把“是否有设备在线”变成 debug 主流程的前置条件。

## 7. 和单通道语音调度的关系

当前 assistant 仍然遵守“同一时刻只允许一个会发声主流程运行”的规则。

因此：

- 如果真实设备上的主流程正在播报，Dashboard 送入的文本会被拒绝
- 如果 Dashboard 刚送入了一条文本，新的设备 ASR 也会因为通道忙而被忽略

这个入口是主流程的一部分，不是绕过主流程的旁路。
